dva与redux一些简单的介绍

目录

  1. redux
  2. redux-saga
  3. dva
  4. fetch – 待补充

为什么使用redux

组件会发生三种通信。react只提供了一种通信手段：传参（可以理解成父子组件间的通信）

向子组件发消息

向父组件发消息

向其他组件发消息

  • 在React中，React组件之间的关系为从属关系，将React组件的关系类比成父子关系。所以数据必须是由一方传到另一方，然而子组件的通信则必须通过父组件进行传递。随着项目的庞大与复杂，这种方式严重影响了代码的可读性与性能。
  • 当然redux使得状态变得可预测在一定程度上避免了bug滋生的温床。

redux的准则

  • 单一数据源。
    整个应用状态，都应该被存储在单一store的对象树中。
  • 只读状态。
    唯一可以修改状态的方式，就是发送（dispatch）一个动作（Action），通俗来讲，就是说只有getter，没有setter。
  • 使用纯函数去修改状态。
    戳这里了解

redux的Action、Reducer和Store

  • Action可以理解成用户的一个交互事件。
  • Reducer就是根据Action来进行更新state。
  • Store用于管理状态的对象，也可以理解为数据库。

redux数据流向

  • 用户对页面操作就会产生一个action事件，然后通过redux的中间插件（redux-saga等）发送异步请求，然后根据返回的结果在reducer中进行判断操作等改变state。

    界面 => action => reducer => store => react => virtual dom => 界面

  • redux将前端合理拆分成不同的层，有关展示为一层，有关事件为一层，有关请求与数据处理又是一层，区别于之前前端开发臃肿的代码结构层级，再加上es6、es7的语法使得前端工程化的形式更加趋向于java。

  • 使用redux，设计更加重要尤其对于复杂庞大的系统，不同的文件，文件代码的功能等等。

React组件的写法

React Component有3种定义方式，分别是React.createClass, class和Stateless Functional Component。推荐尽量使用最后一种，保持简洁和无状态。没有 this 作用域，用过的人都知道这种写法有多好。

####redux-saga

简介

redux-saga是redux的一个中间件，主要集中处理react架构中的异步处理工作，被定义为generator(ES6)的形式，采用监听的形式进行工作。

安装

第二种是Facebook去年年底出的包管理工具（目前可能存在一些小bug），antd也有封装了yarn的脚手架工具

npm install --save redux-saga

yarn add redux-saga

方法（自行了解）
  • put（产生一个 action）
  • select（获取state值）
  • call（阻塞地调用一个函数，个人理解：同步请求）
  • fork（非阻塞地调用一个函数，个人理解：异步请求）
  • take（监听且只监听一次 action）
  • delay（延迟）
  • race（只处理最先完成的任务）

2 监听事件，发送请求

3 reducer

dva

dva是对redux-saga的⼀层浅封装，对redux的概念做了简化。

使用dva

tnpm install dva-cli -g

dva new dva-quickstart

dva数据流向

数据的改变发生通常是通过用户交互行为或者浏览器行为（如路由跳转等）触发的，当此类行为会改变数据的时候可以通过 dispatch 发起一个 action，如果是同步行为会直接通过 Reducers 改变 State ，如果是异步行为（副作用）会先触发 Effects 然后流向 Reducers 最终改变 State，所以在 dva 中，数据流向非常清晰简明，并且思路基本跟开源社区保持一致（也是来自于开源社区）。

dva API 具体
  • app = dva(opts)
  • app.use
  • app.model(model)
  • app.router(({ history, app } => RouterConfig)
  • app.start(selector?)
项目结构

model

dva中的每个model，实际上都是普通的JavaScript对象，包含

  1. namespace：该model的名称

  2. state：state状态的对象

  3. subscriptions： 订阅

  • effects：异步请求，业务逻辑操作

  • reducers：处理同步操作，唯一的一个可以修改state的地方

dva将store、reducer、effect（请求中间插件）结合在model中，一方面减少了文件数量，另一方面减少了层级使得项目更清晰更容易维护。

组件的类别
  • components主要存放项目的各个组件，纯展示的UI组件，没有逻辑与数据处理。
  • routers中存路由组件，连接路由，还有事件的action方法，属于最顶层组件。
    以上组件、文件存放纯属个人喜好，可以根据不同的想法有所不同。
dva的一个事件触发到结束

1 组件 action，用户触发事件

2 监听事件，发送请求

3 reducer

  • 从这里可以看出dva与redux-saga数据流非常相似。dva延续了redux-saga的generator的形式，使用异步的方式写同步代码。并且基本继承了redux-saga的属性和方法。
  • dva的每个函数都封装了redux-saga的generator方法。
  • dva每新增一个model都要在index.js文件中引入，每新增一个顶层组件都要在router.js中配路由
dva-loading

使用dva-loading后就不用在effect中重复地写yield put({type: 'showloading',payload: { loading: true }});。只要在页面中引入loading。

1 项目中引入dva-loading，并加入index.js

2 把loading字段传到组件里

dva-loading存在一定局限性，首先是loading效果是覆盖到整个页面，有些小组件需要单独的loading效果就无法满足，其次想给组件添加dva-loading，必须为其配上独有的model，这两者要一一对应。

router路由的优化

  • 由于dva最初教程例子的路由配置实例是引入组件，并给该组件配置路由信息。这样一来整个router.js的代码显得不够优雅，如图：

  • 有不少业务场景下，我们可能会定义出很多个model，但并不需要在应用启动的时候就全部加载，比较典型的是各类管理控制台。如果每个功能页面是通过路由切换，互相之间没有关系的话，通常会使用webpack的require.ensure来做代码模块的懒加载。

关于model

  • select： 在generator函数中，modelA可以通过select的方法获取modelB的state状态值，这是非常好用的方法

    const valueA = yield select(({ modelB }) => modelB.valueA);

  • put： put的用法和组件中的dispatch方法一样，都是调用一个方法，区别是put是generator定义的方法。

    yield put({ type: 'functionA', payload: { keyA: valueA } });

dispatch({ type: 'functuinA', payload: { keyA: valueA } });

  • call: 执行异步函数，通常用于发送请求，当然也可以同时发送两个请求

    const { result1 } = yield call(service1, parse(params1));

const [result1, result2]  = yield [
  call(service1, param1),
  call(service2, param2)
]

  • take： 可以使用take操作来监听action。take也可以用来处理复杂业务下的操作，比如跨model的方法调用等等，很强大。

不一一赘述了，以上都是常用的方法，想了解更多详情点我

fetch

  • 用过fetch的人都不想用ajax，后者感觉有点low（嘿嘿嘿）。目前本人也在学习中，有些地方感觉和promise很相似。不支持IE。
  • dva、antd以及淘宝其他项目都推荐使用fetch处理异步请求，上手不难，想掌握好很难。

结语

antd、dva以及其延伸的出来的工具方法真的非常强大，而且官网的例子很详细，多多使用，多多对比，多多思考就能掌握好。
